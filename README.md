## 📝 Document Your Experience

### Game Purpose

Game Glitch Investigator is a number guessing game built with Streamlit. The player tries to guess a secret number within a limited number of attempts, with hints guiding them higher or lower after each wrong guess. The game supports three difficulty levels:

- Easy — Guess a number between 1 and 20, with 6 attempts allowed
- Normal — Guess a number between 1 and 100, with 8 attempts allowed
- Hard — Guess a number between 1 and 500, with only 5 attempts allowed

The game was originally generated by an AI but shipped with multiple bugs that made it unplayable. This project involved identifying, diagnosing, and fixing those bugs using VS Code Copilot, then refactoring the logic into a separate module and verifying everything with automated pytest tests.

---

### Bugs Found

Bug 1 — Inverted Hints (Higher/Lower Swapped)

The hint messages inside `check_guess()` were completely backwards. When a player's guess was too high, the game said "Go HIGHER!" — pushing them further in the wrong direction. When the guess was too low, it said "Go LOWER!" — again pointing the wrong way. This made it impossible to win using the hints. For example, entering 1 showed "Go LOWER" and entering 100 showed "Go HIGHER", which is the exact opposite of correct behaviour.

Bug 2 — Negative Scoring

The `update_score()` function contained two scoring errors. First, a stray `attempt_number % 2 == 0` condition was *rewarding* +5 points for wrong "Too High" guesses on even-numbered attempts — wrong guesses should never earn points. Second, there was no floor on the score, so repeated wrong guesses kept deducting points until the score went deeply negative (e.g., a final score of -45).

Bug 3 — New Game Button Not Resetting Properly

Clicking "New Game" after a game over did not properly restart the game. The error message stayed on screen, attempts did not reset, and new input was blocked. The root cause was that `status` was never reset — so `st.stop()` fired immediately on every rerun, blocking the new game. Additionally, `score` and `history` were never cleared, and the new secret was always drawn from 1–100 regardless of the selected difficulty.

Bug 4 — Difficulty Display and Range Issues

Multiple sub-bugs were found related to difficulty settings. The "Make a guess" section always showed "Guess a number between 1 and 100" because the range was hardcoded as a string instead of using the actual `low` and `high` values. The attempts counter showed wrong numbers (including negatives) because `attempts` was initialized to `1` instead of `0`. On even-numbered attempts, the secret was converted to a string, causing lexicographic comparison — meaning `"9" > "10"` evaluated as `True`, producing completely wrong hints. Additionally, Hard difficulty used a range of `(1, 50)`, which is actually narrower and easier than Normal's `(1, 100)`.

Bug 5 — Secret Number Ignoring Difficulty on Switch

The `if "secret" not in st.session_state` guard only runs once on the very first page load, using whatever difficulty was active at that moment (typically "Normal", range 1–100). After that, the key always exists in session state, so switching to Easy or Hard never regenerated the secret. This is why the secret could be 45 even in Easy mode, where the range should only go up to 20. This bug was missed by Copilot on the first fix attempt and was only caught by actually playing the game and observing the out-of-range secret.

---

### Fixes Applied

Bug 1 Fix: The two return messages inside `check_guess()` in `app.py` were swapped so they now correctly say "Go LOWER!" when the guess is too high, and "Go HIGHER!" when the guess is too low.

Bug 2 Fix: The `attempt_number % 2` condition was removed entirely. Both "Too High" and "Too Low" wrong outcomes now consistently deduct 5 points, and `max(0, ...)` was added to floor the score at 0 so it can never go negative.

Bug 3 Fix: The new game block was updated to reset all five session state variables — `status` back to `"playing"`, `score` to `0`, `history` to `[]`, `attempts` to `0`, and the secret number now uses the correct difficulty range from `get_range_for_difficulty()`.

Bug 4 Fix: The range display in `st.info` now dynamically uses `{low}` and `{high}`. The `attempts` initialization was changed to `0`. The string type-casting of the secret was removed entirely so the numeric secret is always passed to `check_guess()`. Hard difficulty range was changed to `(1, 500)` to make it genuinely harder than Normal.

Bug 5 Fix: Difficulty tracking was added to session state. On every render, if the stored difficulty does not match the currently selected difficulty, the game fully resets — generating a new secret within the correct range and clearing all state variables.

---

### Refactor & Tests

All game logic was moved from `app.py` into a dedicated `logic_utils.py` module. The following four functions are now implemented and importable:

- `get_range_for_difficulty(difficulty)` — returns the correct number range for the selected difficulty
- `parse_guess(raw)` — validates and parses user input
- `check_guess(guess, secret)` — compares guess to secret and returns outcome string
- `update_score(current_score, outcome, attempt_number)` — calculates updated score

A `conftest.py` file was added to the `tests/` folder to ensure pytest can correctly find and import `logic_utils` from the project root.
